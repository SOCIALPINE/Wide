# wide — 벤치마크 (BENCH.md)

> "빠르다"는 주장은 측정으로만 정직하다. 이 문서는 wide의 세 실행 경로(트리워커·바이트코드 VM·
> Cranelift JIT)의 실측 수치와, *무엇이 빠르고 무엇이 아직 아닌지*를 그대로 기록한다.

## 측정 방법 (재현: `python bench/run.py`)

- **in-process 시간** — `wide --time`이 stderr로 찍는 실행 시간(파싱+검사+실행). **프로세스 기동
  제외** — 이 머신에선 안티바이러스의 exe 스캔이 실행마다 ~350ms(벽시계)·CPU 0을 붙여 벽시계
  측정을 오염시킨다(정직: 그래서 기동을 빼고 잰다. 원시 벽시계 = 아래 수치 + 기동).
- Python·C·Rust 비교도 같은 방식(각 프로그램이 스스로 계측 구간을 재서 stderr로 보고,
  `bench/native/`). C는 `gcc -O2`, Rust는 `rustc -O`.
- 각 3회 실행 중 최솟값. release 빌드(opt-level 3). 트리워커 수치는 실행 간 편차가 ±20%쯤 있다
  (백그라운드 부하) — 배수 비교가 목적이므로 그대로 둔다.
- 환경: AMD Ryzen 5 7535HS · 16GB · Windows 11 · rustc 1.94 · gcc 15.2 · cranelift 0.133
  (opt_level=speed) · Python 3.11. 측정일 2026-07-04, wide v0.52.

## 결과

| 벤치 | 내용 | 트리워커 | VM | **JIT** | Python | C -O2 | Rust -O |
|------|------|---------:|---:|--------:|-------:|------:|--------:|
| fib.wide | fib(30) 재귀 (호출 270만 회) | 6,375 ms | 1,339 ms | **7.1 ms** | 135 ms | 1.0 ms | 2.1 ms |
| loop.wide | 1,000만 회 정수 while 루프 | 6,747 ms | 5,397 ms | **6.5 ms** | 920 ms | 2.0 ms³ | 5.3 ms³ |
| floatloop.wide | 1,000만 회 f64 while 루프 | 4,945 ms | 4,664 ms | **10.0 ms** | 897 ms | 27.0 ms³ | 31.5 ms³ |
| matmul.wide | 128×128 matmul ×20 (벌크 op) | 29 ms | n/a¹ | 25 ms² | n/a | n/a | n/a |

¹ 텐서는 VM 미지원(트리워커 전용 — 정직한 게이팅).
² matmul은 Rust 백엔드(멀티코어)에서 돌아 JIT과 무관 — 벌크 연산은 JIT 없이도 빠르다(NumPy 모델).
³ **방법론 비대칭 주의**: C/Rust 루프에는 컴파일러가 루프 전체를 닫힌 식(가우스 공식)으로 접는 것을
  막는 배리어(`volatile`/`black_box`)가 필요했고, 그 배리어 자체가 비용이다(배리어 없는 Rust 정수
  루프는 0 ms로 접혔다). wide JIT의 루프에는 그런 배리어가 없다 — 그래서 루프 벤치에서 JIT이 C/Rust보다
  빨라 *보이는* 것은 공정한 비교가 아니다. **배리어가 필요 없는 fib가 가장 공정한 비교다.**

## 읽는 법 (정직한 해석)

- **가장 공정한 비교는 fib(30)**: wide JIT 7.1 ms — **C의 7×, Rust의 3.4× 안쪽, Python보다 19× 빠름,
  자기 인터프리터보다 ~900×.** "동적 타입 + 네이티브 속도" 주장이 *수치 서브셋(i64+f64)에 한해*
  실측으로 성립한다.
- **JIT 적용 범위가 아직 좁다** — 수치 전용(문자열·컬렉션·텐서 폴백), 함수당 단일 ABI(i64 또는 f64,
  교차 ABI 호출 폴백), 나눗셈 제외(/0 파리티). 진짜 타입특화(호출 시그니처별 컴파일)는 다음.
- **C/Rust와의 남은 격차**(fib 기준 3~7×)의 주 원인: Cranelift는 GCC/LLVM만큼 공격적으로 인라인하지
  않고, wide JIT은 아직 함수 단위 최적화뿐이다. 격차를 좁히는 것보다 *어디서 왜 벌어지는지 보이게
  하는 것*이 이 언어의 우선순위다.
- **VM은 트리워커 대비 호출-heavy에서 ~3.5×, 순수 루프에선 근소.** VM의 존재 의의는 속도보다
  JIT(3단계)로 가는 중간 표현 + 파리티 검증대다.
- **matmul은 이미 빠르다** — 언어 속도와 무관하게 벌크 op가 Rust(멀티코어)로 돌기 때문(NumPy 모델).
  AI 워크로드의 속도는 JIT이 아니라 백엔드(다음: 실GPU)가 결정한다.
- opt_level=speed가 중요했다: 기본(none)은 변수를 스택 슬롯에 스필해 루프가 ~60× 느렸다.

## 실GPU 백엔드 (v0.50, `--features gpu` — wgpu, AMD Radeon iGPU에서 측정)

| matmul | CPU 멀티코어 | **GPU (wgpu)** | 승자 |
|--------|------------:|---------------:|------|
| 512×512 ×10 | 603 ms | 1,150 ms | **CPU** (전송/eager D2H 오버헤드 지배) |
| 1024×1024 ×5 | 2,410 ms | **838 ms** | **GPU** (2.9×) |

- 교차점이 512~1024 사이 — **작은 행렬은 GPU가 오히려 느리다.** 원인: 결과마다 eager D2H 읽기
  (autodiff/print가 호스트 데이터를 읽으므로) + naive WGSL 셰이더(타일링 없음) + 통합 GPU.
  wide는 이 비용을 조명으로 그대로 보여준다("D2H result N B") — *이게 언어의 논지다*.
- 체인의 재업로드는 실제로 0(버퍼 캐시, "resident on device — no transfer" 조명).
- 개선 여지(정직): lazy D2H(결과를 쓸 때만 다운로드), 타일드 셰이더, 전용 GPU.

## 벤치 추가하는 법
